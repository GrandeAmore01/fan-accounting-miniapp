# 用户头像昵称云端资料迁移清单

## 一、目的与边界

- 为真实微信用户保存自定义昵称和 CloudBase 云存储头像 `fileID`。
- 头像文件本身保存在 CloudBase 云存储，不写入 TiDB。
- 不修改 `user_id`、`openid`、现有唯一索引或外键。
- 不迁移 `local-user` 数据。

## 二、TiDB 负责人先执行预检

```sql
USE fan_accounting;

SHOW CREATE TABLE users;
SHOW COLUMNS FROM users LIKE 'display_name';
SHOW COLUMNS FROM users LIKE 'avatar_file_id';
SELECT COUNT(*) AS users_total FROM users;
```

请把完整结果返回给代码负责人审核。字段已经存在或线上结构与仓库声明不一致时，不要直接执行迁移。

## 三、确认字段不存在后执行

```sql
USE fan_accounting;

ALTER TABLE users
  ADD COLUMN display_name VARCHAR(40) COLLATE utf8mb4_unicode_ci NULL AFTER openid;

ALTER TABLE users
  ADD COLUMN avatar_file_id VARCHAR(500) COLLATE utf8mb4_bin NULL AFTER display_name;
```

风险等级：低。该操作只新增两个允许 `NULL` 的字段，不更新或删除现有用户数据。

## 四、执行后验证

```sql
USE fan_accounting;

SHOW CREATE TABLE users;
SHOW COLUMNS FROM users LIKE 'display_name';
SHOW COLUMNS FROM users LIKE 'avatar_file_id';
SELECT COUNT(*) AS users_total FROM users;
```

预期：

- `display_name` 为 `VARCHAR(40)`，允许 `NULL`。
- `avatar_file_id` 为 `VARCHAR(500)`，允许 `NULL`。
- `users_total` 与执行前一致。
- `user_id`、`openid`、`uk_users_openid` 均未改变。

## 五、迁移后的部署顺序

1. 合并头像昵称功能 PR。
2. 手动触发 GitHub Actions，使用新 Git commit SHA 构建镜像。
3. 在 CloudBase 创建新版本，填写新的 SHA 镜像地址。
4. 保留当前稳定版本，不复用旧镜像标签。
5. 验证 `GET /api/profile` 和 `PUT /api/profile`。
6. 真机验证昵称保存、头像上传、重新进入、清缓存和重启后的恢复。
