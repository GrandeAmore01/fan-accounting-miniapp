# 数据结构与联动逻辑说明

本文档给需要改公共 service、统计逻辑或跨页面联动的同学使用。普通页面样式和局部交互可以只看 `04_开发交接分工说明.md`；如果要新增筛选、导出、分类预算、自定义藏品、年度报告等功能，建议先读完这份。

## 1. 当前数据保存方式

当前前端主要使用本地缓存，统一入口在 `services/storageService.js`。

本地缓存 key 格式：

```js
fan_accounting_${userId}
```

当前默认用户是 `local-user`。一个用户的数据结构大致如下：

```js
{
  expenses: [],
  budgets: [],
  userCollections: [],
  userStages: [],
  profile: {
    nickname: '',
    loginStatus: false
  }
}
```

读写公共数据时优先使用：

- `storageService.getCollection(userId, collectionName)`
- `storageService.setCollection(userId, collectionName, value)`

不要在页面里直接拼缓存 key。这样以后接后端或云开发时，改动范围会更小。

## 2. 消费记录 expenses

消费记录是预算、藏品、舞台联动的中心数据。核心逻辑在 `services/expenseService.js`，后端预留模型在 `server/src/utils/expenseModel.js`。

单条消费记录主要字段：

```js
{
  expenseId: 'expense_...',
  category: 'meet',
  subType: 'concert',
  itemName: '',
  amount: 0,
  quantity: 1,
  date: '2026-07-05',
  paymentMethod: '',
  seat: '',
  location: '',
  remark: '',
  images: [],
  fees: {
    premium: 0,
    travel: 0,
    hotel: 0,
    rental: 0,
    other: 0,
    shipping: 0
  },
  outfieldOnly: false,
  includeInTotal: true,
  collectionId: '',
  stageId: '',
  stageDate: '',
  priceTier: ''
}
```

页面展示时，`expenseService.enrichExpense` 会额外计算：

- `categoryName`：大类/小类展示名。
- `baseAmount`：基础金额，计算方式是 `amount * quantity`。
- `totalAmount`：实际总金额，基础金额加全部附加费用。
- `includedAmount`：计入预算和总消费的金额。
- `feeTags`：有金额的附加费用标签。

## 3. 消费金额计算规则

消费金额不要在页面里重新手写计算，优先复用 `expenseService` 里的函数。

当前规则：

- `baseAmount = amount * quantity`
- `totalAmount = baseAmount + premium + travel + hotel + rental + other + shipping`
- `includeInTotal === false` 时，`includedAmount = 0`
- 演唱会消费如果 `outfieldOnly === true`，只把门票/基础金额计入预算，附加费用不计入。
- 其他类型如果 `outfieldOnly === true`，会被视为不计入总消费。

做高级筛选、导出账单、预算统计时要先确认用哪个金额：

- 实际花了多少钱：用 `totalAmount`
- 计入预算多少钱：用 `includedAmount`
- 商品或门票基础价：用 `baseAmount`

## 4. 消费分类

消费分类来自 `data/expenseTypes.js`。

当前大类：

- `meet`：见面，子类包括 `concert`、`festival`、`activity`、`other_meet`
- `album`：实体专辑
- `goods`：周边
- `business`：商务

如果新增分类，至少要检查：

- `data/expenseTypes.js`
- `pages/expenses/index.js`
- `services/expenseService.js`
- `services/budgetService.js`
- 和藏品、舞台生成消费记录时的分类映射

不建议每位同学各自随意新增分类 id，否则预算统计和导出会变得不统一。

## 5. 预算 budgets

预算逻辑在 `services/budgetService.js`。

单条预算主要字段：

```js
{
  budgetId: 'budget_month_2026-07',
  budgetType: 'month',
  amount: 1000,
  period: '2026-07',
  threshold: 0.8
}
```

字段说明：

- `budgetType`：`month` 或 `year`
- `period`：月预算使用 `YYYY-MM`，年预算使用 `YYYY`
- `amount`：预算金额
- `threshold`：提醒阈值，当前默认 `0.8`

预算统计当前依赖 `expenseService.listExpenses()`，并只统计：

- `includeInTotal !== false`
- 消费日期落在当前预算周期内

如果同学林要做分类预算，可以考虑两种方案：

- 在 `budgets` 中新增字段，例如 `categoryBudgets: [{ category, amount, threshold }]`
- 新增独立集合，例如 `categoryBudgets`

更推荐第一种，和月/年预算周期绑定会更直观。但改之前要和组内确认，避免影响现有预算保存逻辑。

## 6. 藏品 userCollections

内置藏品数据来自 `data/collections.js`，用户状态存在本地 `userCollections`，业务逻辑在 `services/collectionService.js`。

内置藏品示例字段：

```js
{
  collectionId: 'col_album_001',
  collectionName: '',
  category: 'album',
  referencePrice: 128,
  cover: '',
  description: ''
}
```

用户藏品状态字段：

```js
{
  userId: 'local-user',
  collectionId: 'col_album_001',
  isOwned: true,
  lightTime: '2026-07-05T...',
  expenseId: 'expense_...'
}
```

联动规则：

- 点亮藏品时，先写入或更新 `userCollections`。
- 如果用户选择生成消费记录，会调用 `collectionService.createExpenseFromCollection`。
- 生成成功后，会把消费记录的 `expenseId` 回写到对应 `userCollections` 状态中。
- 取消点亮时，如果存在 `expenseId`，页面会询问是否一起删除关联消费记录。

如果要做自定义藏品，不建议直接改内置 `data/collections.js`。更稳的方案是新增本地集合，例如：

```js
customCollections: [
  {
    collectionId: 'custom_...',
    collectionName: '',
    category: '',
    referencePrice: 0,
    actualPrice: 0,
    cover: '',
    description: '',
    wishStatus: 'owned'
  }
]
```

然后在 `collectionService.listCollections()` 中把内置藏品和自定义藏品合并展示。

## 7. 舞台 userStages

内置舞台数据来自 `data/stages.js`，用户点亮状态存在本地 `userStages`，业务逻辑在 `services/stageService.js`。

内置舞台示例字段：

```js
{
  stageId: 'stage_2024_001',
  stageName: '',
  stageType: 'concert',
  year: 2024,
  date: '2024-08-17',
  city: '',
  venue: '',
  location: '',
  albumId: '',
  priceTiers: [480, 680, 980],
  songList: []
}
```

用户舞台状态字段：

```js
{
  userId: 'local-user',
  stageId: 'stage_2024_001',
  isLighted: true,
  lightTime: '2026-07-05T...',
  expenseId: 'expense_...'
}
```

联动规则：

- 点亮舞台时，先写入或更新 `userStages`。
- 如果用户选择生成消费记录，会调用 `stageService.createExpenseFromStage`。
- 消费记录会写入 `stageId`、`stageDate`、`priceTier`。
- 生成成功后，会把消费记录的 `expenseId` 回写到对应 `userStages` 状态中。
- 取消点亮时，如果存在 `expenseId`，页面会询问是否一起删除关联消费记录。

如果要做舞台详情、照片、同行人、座位备注等，建议新增用户侧详情集合，不要直接改内置舞台数据：

```js
stageNotes: [
  {
    stageId: 'stage_2024_001',
    seat: '',
    companions: '',
    photos: [],
    note: '',
    actualTicketPrice: 0
  }
]
```

这样不会影响 `data/stages.js` 的公共场次数据。

## 8. 藏品、舞台、消费的联动关系

当前跨页面联动主要靠下面几个 id：

- `expenses.collectionId` 关联 `collections.collectionId`
- `userCollections.expenseId` 关联 `expenses.expenseId`
- `expenses.stageId` 关联 `stages.stageId`
- `userStages.expenseId` 关联 `expenses.expenseId`

可以理解成：

```text
藏品点亮状态 userCollections
  -> expenseId
  -> 消费记录 expenses
  -> collectionId
  -> 内置藏品 collections

舞台点亮状态 userStages
  -> expenseId
  -> 消费记录 expenses
  -> stageId
  -> 内置舞台 stages
```

新增详情页、年度报告、导出账单时，尽量复用这些 id，而不是用名称做匹配。名称可能重复或后续改文案，id 更稳定。

## 9. 页面功能和 service 的关系

建议按下面原则写代码：

- 页面负责展示、表单状态、用户点击事件。
- service 负责数据读写、筛选、统计、金额计算、联动生成记录。
- data 目录只放公共静态数据，不放用户个人状态。

例如：

- 高同学做消费高级筛选：页面收集筛选条件，`expenseService` 负责真正筛选。
- 林同学做预算分析：页面展示图表，`budgetService` 负责统计结果。
- 于同学做自定义藏品：页面做新增表单，`collectionService` 负责合并内置和自定义数据。
- 杨同学做年度报告：页面展示报告，`stageService` 负责计算场次、城市、歌单，必要时再从 `expenseService` 取花费。

## 10. 常见改动建议

### 高级筛选

建议筛选对象统一使用 `expenseService.listExpenses()` 返回的 enriched 数据，因为里面已经有 `includedAmount`、`totalAmount`、`categoryName`。

筛选条件建议字段：

```js
{
  keyword: '',
  category: 'all',
  startDate: '',
  endDate: '',
  minAmount: '',
  maxAmount: '',
  amountType: 'included',
  includeInTotal: 'all',
  paymentMethod: 'all'
}
```

### 导出账单

建议导出字段：

- 日期
- 分类
- 项目名称
- 基础金额
- 实际总金额
- 计入预算金额
- 支付方式
- 地点/座位
- 是否计入预算
- 备注

CSV 要注意逗号、换行和引号转义。

### 预算历史

预算历史可以从 `budgets` 读取预算设置，再按月份扫描 `expenses` 计算实际消费。不要只展示有预算的月份，也可以考虑展示有消费但没设置预算的月份。

### 自定义藏品

自定义藏品要有自己的 id 前缀，例如 `custom_collection_`，避免和内置 `collectionId` 冲突。

### 年度报告

年度花费不要只按舞台 `date` 估算，优先查 `expenses` 中带 `stageId` 的消费记录；这样票价、加价、路费等更准确。

## 11. 提交前自查

改公共逻辑前后建议检查：

- 新增消费、编辑消费、删除消费是否还能正常工作。
- 藏品点亮生成消费记录后，取消点亮是否还能处理关联记录。
- 舞台点亮生成消费记录后，取消点亮是否还能处理关联记录。
- 预算统计是否仍然只统计应计入预算的记录。
- 空数据时页面是否有正常提示。
- 本地缓存旧数据缺字段时，页面是否会报错。

如果新增字段，尽量在 service 的 normalize 或 list 方法里给默认值，避免旧缓存数据导致页面崩溃。
