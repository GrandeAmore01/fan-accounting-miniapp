# 开发交接分工说明

本文档用于把当前小程序已有功能、代码入口和后续分工交接给其他开发同学。下面列出的增强方向是建议，不是硬性任务清单；每位同学可以结合页面现状、答辩重点和自己的想法做取舍。

如果后续要修改公共 service、数据结构或跨页面联动，请同时阅读 `docs/05_数据结构与联动逻辑说明.md`。

## 1. 项目当前状态

项目是微信小程序，主要页面都在 `pages/` 下，业务逻辑集中在 `services/`，公共静态数据在 `data/`。

当前已完成的基础能力包括：

- 消费记录：新增、编辑、删除、分类、关键词搜索、费用计算、是否计入总消费、图片选择、与舞台/藏品联动生成消费记录。
- 预算统计：月预算、年预算保存，预算进度，分类统计，月度趋势基础展示。
- 藏品图鉴：内置藏品列表，分类/状态/关键词筛选，点亮和取消点亮，点亮后可生成消费记录。
- 舞台记录：舞台列表，筛选，点亮和取消点亮，点亮后可生成消费记录，歌单统计，专辑进度，见面回忆页。
- 本地存储：当前主要使用 `wx.getStorageSync` / `wx.setStorageSync` 保存本地用户数据，服务层统一通过 `storageService` 读写。

## 3. 同学高：消费记录高级功能升级

### 当前基础

消费记录页已经支持基础记账、编辑、删除、分类筛选、关键词搜索、费用计算、图片选择、是否计入总消费，以及和舞台/藏品的联动。也就是说当前已经能完成“记录一笔消费”的主流程。

### 建议目标

把消费记录从“能记账”升级为“好管理”。可以优先围绕列表管理、查找、导出和详情查看来做。

### 可选增强方向

- 高级筛选：开始日期、结束日期、金额区间、是否计入预算、支付方式筛选。
- 排序能力：按时间、金额、分类排序，也可以增加升序/降序切换。
- 消费详情：增加详情弹窗或详情页，展示图片、费用拆分、备注、关联舞台/藏品等完整信息。
- 批量操作：批量删除、批量设置是否计入预算。
- 导出账单：支持 CSV 或纯文本格式，便于展示“可导出账单”的完整管理能力。
- 状态优化：空状态、错误提示、加载状态、保存成功/失败提示可以做得更清楚。

### 建议改动位置

- 页面状态和事件：`pages/expenses/index.js`
- 列表、筛选区、弹窗结构：`pages/expenses/index.wxml`
- 筛选、批量、详情弹窗样式：`pages/expenses/index.wxss`
- 如果筛选/排序逻辑希望复用，可扩展 `services/expenseService.js`，例如增强 `filterExpenseList` 或新增导出函数。

### 注意事项

- 费用字段里同时有 `amount`、`quantity`、`fees`、`totalAmount`、`includedAmount`，筛选金额时要先确认筛选的是实际总金额还是计入预算金额。
- `includeInTotal` 会影响预算统计，不要把“不计入总消费”的记录误算进预算。
- 舞台和藏品生成的消费记录会带 `stageId` 或 `collectionId`，详情页可以展示这些关联信息。

## 4. 同学林：预算统计可视化升级

### 当前基础

预算页已经支持月/年预算保存、预算进度、提醒阈值、分类统计和基础月度趋势。当前页面已经能回答“花了多少、还剩多少、有没有超预算”。

### 建议目标

把统计页升级成分析页，让答辩时可以讲“预算分析、预警、历史和分类预算”。

### 可选增强方向

- 图表展示：分类占比图、月度趋势图、预算进度环。
- 预算预警记录：超过 80%、100% 时给出不同提醒，也可以保留历史预警。
- 本月消费分析文案：例如最高消费分类、比上月多/少多少、预算使用建议。
- 预算历史列表：查看过去每个月预算和实际消费。
- 分类预算：给演唱会、周边、专辑等分类分别设置预算，而不只是总预算。

### 建议改动位置

- 页面状态和刷新逻辑：`pages/budget/index.js`
- 统计卡片、趋势区、分析文案、历史列表：`pages/budget/index.wxml`
- 图表和预算进度视觉：`pages/budget/index.wxss`
- 统计计算与历史数据：`services/budgetService.js`
- 分类名称和分类来源可参考：`data/expenseTypes.js`

### 注意事项

- 预算统计当前依赖 `expenseService.listExpenses()`，只统计 `includeInTotal !== false` 且日期落在周期内的记录。
- 如果做分类预算，需要设计新的本地存储结构，建议放在 `budgets` 里扩展字段或单独新增 collection，先和组内确认。
- 图表可以先用 WXML/CSS 做轻量柱状、环形或进度条；不一定必须引入复杂图表库。

## 5. 同学于：藏品图鉴内容与详情升级

### 当前基础

藏品图鉴页已经支持内置藏品列表、分类筛选、状态筛选、关键词搜索、点亮、取消点亮，并能在点亮后生成对应消费记录。

### 建议目标

把图鉴从“点亮列表”升级成“个人藏品管理系统”。

### 可选增强方向

- 藏品详情：增加详情页或详情弹窗，展示图片、价格、发行时间、购买状态、备注。
- 用户自定义藏品：允许用户新增自己的藏品，不只依赖内置数据。
- 图片管理：上传或更换藏品图片。
- 收藏进度分组：专辑、小卡、杂志、周边分别统计完成度。
- 愿望清单：标记想买但还没买的藏品。
- 价格维护：官方价、实际购入价、溢价金额。

### 建议改动位置

- 页面状态和操作：`pages/collections/index.js`
- 列表、详情弹窗、新增表单：`pages/collections/index.wxml`
- 卡片、图片墙、进度分组样式：`pages/collections/index.wxss`
- 藏品状态和新增藏品逻辑：`services/collectionService.js`
- 内置藏品数据：`data/collections.js`
- 分类数据：`data/categories.js`

### 注意事项

- 当前内置藏品来自 `data/collections.js`，用户点亮状态存在本地 `userCollections`。如果新增自定义藏品，建议区分“内置数据”和“用户自定义数据”。
- 点亮生成消费记录时会写入 `expenseId`，取消点亮时可以选择是否删除关联消费记录，不要破坏这条联动关系。
- 如果要支持图片，参考消费记录页的 `wx.chooseMedia` 和 `images` 字段实现。

## 6. 同学杨：舞台回忆与报告升级

### 当前基础

舞台页已经支持演唱会/音乐节筛选、年份筛选、点亮、取消点亮、生成消费记录、歌单统计、专辑进度，并且已有一个回忆页用于展示见面次数、城市/场馆排行等基础报告。

### 建议目标

把舞台记录从“场次列表”升级成“观演回忆和年度报告”。

### 可选增强方向

- 舞台详情：展示场次信息、座位、票价、同行人、照片、备注。
- 我的观演日历：按月份展示看过或将要看的场次。
- 年度回忆报告：今年见了几次、去了几个城市、花了多少钱。
- 城市地图或城市列表统计：去过城市排行、场馆排行。
- 歌单收藏：最常听到的歌、还没听过的歌。
- 回忆图片墙：每场演出可关联照片。

### 建议改动位置

- 舞台列表和点亮操作：`pages/stages/index.js`
- 舞台列表、筛选、详情入口：`pages/stages/index.wxml`
- 回忆报告页面：`pages/memories/index.js`、`pages/memories/index.wxml`
- 样式：`pages/stages/index.wxss`、`pages/memories/index.wxss`
- 舞台统计和回忆报告计算：`services/stageService.js`
- 舞台基础数据：`data/stages.js`
- 年度花费统计可复用消费记录：`services/expenseService.js`

### 注意事项

- 舞台点亮状态存在本地 `userStages`，不是直接改 `data/stages.js`。
- 舞台生成消费记录会记录 `stageId`、`stageDate`、`priceTier`，年度花费报告可以通过这些字段关联计算。
- 如果做日历或年度报告，建议优先复用 `stageService.getMeetMemoryReport`、`getMeetTimeline`、`getSongStats`、`getAlbumProgress` 的已有数据。

## 7. 共同约定

- 每个人从最新 `main` 新建自己的功能分支，不直接在 `main` 上开发。
- 分支命名建议：`feature/gao-expenses`、`feature/lin-budget`、`feature/yu-collections`、`feature/yang-stages-memory`。
- 每个 PR 尽量只包含自己负责模块的功能，避免一个 PR 同时改多个不相关页面。
- 公共文件包括 `app.json`、`app.wxss`、`services/`、`data/`、`server/` 等，修改前先说明原因和影响。
- 提交前至少在微信开发者工具里跑通自己负责页面的主流程。

## 8. 建议验收标准

每位同学交付时可以按下面格式自查：

- 做了哪些功能。
- 改了哪些页面和文件。
- 是否改了公共 service 或 data。
- 是否能和已有消费记录、预算、藏品、舞台联动逻辑共存。
- 是否测试了新增、编辑、删除、筛选、空状态等关键流程。
- 是否有截图或录屏方便组内确认。

## 9. 给同学们的提醒

上面的功能只是建议，不是必须全部完成。大家可以根据自己的时间和想法自由取舍，能做多少就做多少；重点是把自己选择的部分做完整、做顺，而不是堆很多半成品。
