# 模块云端迁移部署说明

本文档用于指导各模块负责人将自己负责的模块逐步迁移到云端。

当前项目采用“分模块逐步上云”的方式：已经迁移完成的模块单独走云端，尚未迁移的模块继续走本地接口。等所有模块都完成云端迁移和测试后，再统一切换全局接口地址。

## 1. 当前项目状态

当前 `services/config.js` 的核心配置为：

```js
module.exports = {
  useBackend: true,
  useStageBackend: true,
  apiBaseUrl: 'http://127.0.0.1:3000/api',
  collectionApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  userId: 'local-user'
};
```

含义：

- `apiBaseUrl`：全局默认接口地址，目前仍指向本地后端，供消费、预算、舞台等未完成云端迁移的模块使用。
- `collectionApiBaseUrl`：藏品图鉴模块专用云端接口地址。
- `userId: 'local-user'`：阶段一调试用用户 ID，后续 OpenID 接入后会替换为真实微信用户身份。

当前阶段目标：

```text
藏品图鉴模块走云端。
其他模块继续走本地。
各模块单独迁移、单独测试。
所有模块都迁移完成后，再统一切全局 apiBaseUrl 到云端。
```

## 3. 重要原则：不要覆盖别人的表

所有人只操作自己负责模块的数据表，不要删除、清空或重建其他模块的数据表。

目前已知表和模块关系：

| 模块 | 公共/业务表 | 用户状态表 |
| --- | --- | --- |
| 藏品图鉴 | `collections` | `user_collections` |
| 用户基础信息 | `users` | - |
| 消费模块 | `expenses` | 按模块设计决定 |
| 预算模块 | `budgets` | 按模块设计决定 |
| 舞台模块 | `stages` | `user_stages` |
| 回忆/备注 | `memories`、`stage_notes` | 按模块设计决定 |

禁止在未确认的情况下执行：

```sql

DELETE FROM collections;

```

建议所有建表语句使用：

```sql

```

如果必须修改公共表或其他同学负责的表，需要先在群里说明并确认。

## 4. 推荐迁移方式

不要直接把全局 `apiBaseUrl` 改成云端。

错误方式：

```js
apiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api'
```

这样会导致所有模块都提前访问云端。如果某个模块的云端表或接口还没有准备好，就会出现 500、503、表不存在等错误。

正确方式：每个模块新增自己的 `xxxApiBaseUrl`。

示例：

```js
module.exports = {
  useBackend: true,
  useStageBackend: true,
  apiBaseUrl: 'http://127.0.0.1:3000/api',
  collectionApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  expenseApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  budgetApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  stageApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  userId: 'local-user'
};
```

每个模块自己的 service 中使用：

```js
const MODULE_API_BASE_URL = config.xxxApiBaseUrl || config.apiBaseUrl;
```

请求时传入：

```js
apiService.request({
  baseUrl: MODULE_API_BASE_URL,
  url: '/your-module-api'
});
```

这样可以做到：

```text
已迁移模块：走云端。
未迁移模块：继续走本地。
模块之间互不影响。
```

## 5. 迁移前准备

### 5.1 拉取最新代码

```bash
git checkout main
git pull origin main
```

### 5.2 新建自己的功能分支

不要直接在 `main` 上改。

```bash
git checkout -b feature/姓名-模块名-cloud
```

示例：

```bash
git checkout -b feature/zhangsan-expenses-cloud
```

### 5.3 明确自己负责的模块

迁移前先确认自己负责的页面、service、后端路由和数据库表。

示例：消费模块

```text
前端页面：pages/expenses
前端 service：services/expenseService.js
后端路由：server/src/routes/expenses.js
数据库表：expenses
```

## 6. TiDB Cloud 数据库操作

### 6.1 进入 SQL Editor

进入 TiDB Cloud，打开：

```text
SQL Editor
```

选择数据库：

```sql
USE fan_accounting;
```

### 6.2 创建自己模块的数据表

只创建或修改自己模块需要的表。

示例：

```sql
CREATE TABLE IF NOT EXISTS expenses (
  id VARCHAR(64) PRIMARY KEY,
  user_id VARCHAR(64) NOT NULL,
  amount DECIMAL(10, 2) NOT NULL,
  category VARCHAR(64),
  remark TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

建表后建议查询确认：

```sql
SHOW TABLES;
```

或：

```sql
SELECT *
FROM expenses
LIMIT 5;
```

### 6.3 表设计建议

公共数据表：

```text
所有用户共用的数据
例如 collections、stages
```

用户状态表：

```text
按 user_id 区分的数据
例如 user_collections、user_stages
```

后续接入 OpenID 后，用户状态表会按真实微信用户隔离。

## 7. 后端接口开发

后端目录：

```text
server
```

路由目录：

```text
server/src/routes
```

每个模块应有自己的路由文件，例如：

```text
server/src/routes/expenses.js
server/src/routes/collections.js
server/src/routes/stages.js
```

后端入口：

```text
server/src/app.js
```

需要在 `app.js` 中挂载路由：

```js
app.use('/api/expenses', expenseRoutes);
```

接口至少应覆盖模块需要的基础能力：

```text
GET：查询
POST：新增
PUT/PATCH：修改
DELETE：删除
```

## 9. CloudBase 云托管部署

注意：这里不是让每个同学都新建一个云托管服务。

我们使用的是同一个 CloudBase 云托管后端服务：

```text
fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com
```

每个模块负责人需要做的是：

```text
1. 在本地完成自己模块的后端接口
2. 在 TiDB Cloud 建好自己模块需要的表
3. 本地测试接口通过
4. 提交 PR
5. PR 合并后，由负责部署的人更新同一个 CloudBase 云托管服务
```

也就是说：

```text
模块负责人不需要各自创建新的 CloudBase 服务。
CloudBase 服务只需要有一个。
模块代码合并后，再统一部署这个服务的最新版 server。
```

进入腾讯云 CloudBase 云托管服务。

部署配置：

```text
分支：main 或自己的测试分支
服务目录：server
启动命令：npm start
```

推荐实际操作方式：

```text
日常开发阶段：
各模块先在本地开发和测试。

PR 合并后：
由组长或指定同学部署 main 分支的 server。

需要提前验证某个模块时：
可以临时部署该模块分支，但要提前在群里说明。
验证完成后再切回 main。
```

部署完成后，先测试：

```text
https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api/health
```

再测试自己模块接口：

```text
https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api/expenses?userId=local-user
```

如果接口报错，优先检查：

- CloudBase 是否部署了最新代码
- TiDB 中对应表是否存在
- 后端 SQL 表名和字段名是否正确
- 环境变量是否配置完整
- 启动命令是否为 `npm start`
- 服务目录是否选择 `server`

## 9.1 谁需要操作 CloudBase

一般情况下，只有负责部署的人需要操作 CloudBase。

模块负责人主要负责：

- 建自己的 TiDB 表
- 写自己的后端路由
- 写自己的前端 service
- 本地测试通过
- 提交 PR
- 告诉部署负责人：自己的模块已合并，可以部署

部署负责人主要负责：

- 进入 CloudBase 云托管
- 选择最新 `main` 分支
- 服务目录选择 `server`
- 启动命令保持 `npm start`
- 部署完成后测试 `/api/health`
- 再通知对应模块负责人测试自己的云端接口

这样可以避免多人同时改同一个云托管配置，降低误操作风险。

## 10. 前端 service 改造

### 10.1 修改 `services/config.js`

不要改全局 `apiBaseUrl` 为云端。

新增自己模块的云端地址。

示例：

```js
module.exports = {
  useBackend: true,
  useStageBackend: true,
  apiBaseUrl: 'http://127.0.0.1:3000/api',
  collectionApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  expenseApiBaseUrl: 'https://fan-accounting-api-279639-5-1451816515.sh.run.tcloudbase.com/api',
  userId: 'local-user'
};
```

### 10.2 修改自己模块的 service

示例：

```js
const apiService = require('./apiService');
const config = require('./config');

const EXPENSE_API_BASE_URL = config.expenseApiBaseUrl || config.apiBaseUrl;

function listExpenses() {
  return apiService.request({
    baseUrl: EXPENSE_API_BASE_URL,
    url: `/expenses${apiService.buildQuery({ userId: config.userId })}`
  });
}
```

### 10.3 不要改其他模块的 service

只改自己负责模块的 service。

例如消费模块不要修改：

```text
collectionService.js
stageService.js
budgetService.js
```

除非已经和对应负责人确认。

## 15. PR 提交要求

每个模块提交 PR 时，需要说明：

- 迁移了哪个模块
- 新增或修改了哪些数据表
- 修改了哪些前端 service
- 修改了哪些后端路由
- 是否修改公共文件
- 是否影响其他模块
- 是否已经在微信开发者工具运行
- 是否完成真机测试

如果修改公共文件，例如：

```text
services/config.js
services/apiService.js
server/src/app.js
```

需要在 PR 中说明原因，并在群里同步。

PR 中不要包含：

```text
.codex/
%SystemDrive%/
临时脚本
本地调试缓存
```
